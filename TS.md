## 1. ОБЩАЯ КОНЦЕПЦИЯ

### 1.0 Краткое описание

Название: FormatHub  
Тип: локальный веб‑сервис и open‑source‑платформа для конвертации, сжатия и базового редактирования файлов различных типов (изображения, аудио, видео, документы, таблицы, презентации, архивы, электронные книги, векторная графика, 3D‑модели, CAD, исходный код и данные, базы данных, образы дисков, пакеты приложений).

Поддержка 3D‑моделей (STL, OBJ, GLTF/GLB, BLEND и др.), CAD (DWG, DXF, STEP), кода и данных (JavaScript/TypeScript, HTML, CSS, JSON, Markdown), баз данных (SQL, SQLite), образов дисков (ISO, IMG, DMG) и пакетов приложений (APK, DEB, RPM, MSI) добавлена в основной объём поддержки. Распределение по этапам реализации (базовые форматы — в MVP, тяжёлые/редкие — в расширенных этапах Roadmap) описано в разделе «Ограничения и предположения» и Roadmap.

Целевая аудитория (базовая):  
Обычные пользователи и технические специалисты, которым нужно быстро конвертировать или сжать файлы без установки тяжёлого специализированного ПО и без передачи данных в облако — запуск одной командой на своём компьютере и работа через браузер.

Режим работы (MVP):  
Сервис запускается локально командой `npm run dev` (или через Docker/скрипт деплоя) и открывается в браузере по адресу `http://localhost:3000`. Вся обработка происходит на локальной машине пользователя; файлы не отправляются на внешние серверы.

В дальнейшем допускаются дополнительные режимы: self‑hosted на сервере организации и потенциальный публичный SaaS с тем же UI и контролируемыми лимитами (см. раздел «Режимы развёртывания» и Roadmap).

### 1.1 Ключевые особенности продукта

- Локальная обработка и офлайн‑режим.  
  Все операции выполняются на стороне пользователя с использованием установленных системных инструментов (FFmpeg, LibreOffice, Ghostscript, pandoc/calibre, Tesseract, libvips‑plugins и др.), что обеспечивает максимальную приватность и отсутствие зависимости от сторонних сервисов.

- Гибкие, но честные лимиты.  
  Отсутствуют искусственные ограничения на количество операций. Базовый технический лимит размера файла задаётся переменной окружения (по умолчанию 2 ГБ на файл) и сопровождается предупреждениями о возможной нагрузке на железо; администратор может изменить лимит под свои нужды.

- Живой прогресс через WebSocket.  
  Все операции (конвертация, сжатие, инструменты, batch) сопровождаются WebSocket‑стримом статуса: состояние задачи, процент, скорость, ориентировочное время завершения (ETA), а также возможностью отмены задач и батчей. Для операций без нативного прогресса используется контролируемая эмуляция (см. раздел «Эмуляция прогресса»).

- Интеллектуальная фильтрация форматов.  
  Целевые форматы автоматически фильтруются на основе исходного типа файла, поддерживаемых маршрутов конвертации и возможностей системы (systemCapabilities). Пользователь видит только реально доступные и логичные варианты, неподдерживаемые комбинации конвертации блокируются с явной ошибкой (`UNSUPPORTED_CONVERSION`).

- Визуальная группировка и поиск.  
  Все форматы визуально сгруппированы по категориям (images, video, audio, documents, spreadsheets, presentations, archives, ebooks, vector, 3d, cad, code, database, disk, apps и др.), имеют цветовую кодировку. Поиск формата работает с автодополнением; неподходящие варианты отображаются «приглушёнными» и в конце списка.

- Современный интерфейс.  
  Liquid Glass‑дизайн (эффект матового стекла) с аккуратными анимированными градиентами по бокам, тёмная/светлая тема, переключение языков (русский/английский), плавные анимации (Framer Motion или аналог), которые могут быть отключены в режиме низкой производительности.

- Умная работа с производительностью.  
  Glitch‑анимация ошибок (с «тряской» экрана), FPS‑тест и флаг `lowPerformance` для слабых устройств, уменьшение частоты обновления прогресса, отключение тяжёлых визуальных эффектов (`backdrop-blur`, сложные градиенты, «финишная» анимация прогресса) при необходимости.

- История операций.  
  Страница `/history` с возможностью просмотра, скачивания и удаления файлов. История хранится до ручного удаления или очистки, с сохранением параметров операций и путей к результатам. При запуске сервера битые записи (файла по пути нет) автоматически удаляются из `history.json`.

- Автопроверка зависимостей.  
  При старте сервера выполняется проверка наличия системных инструментов (FFmpeg, LibreOffice, Ghostscript, pandoc/calibre, Tesseract, libvips‑plugins и др.). Актуальный статус отдаётся через `GET /api/system/capabilities` и используется фронтендом для отображения доступности функций и модальных инструкций по установке.

### 1.2 Целевая аудитория и отрасли

#### 1.2.1 Массовые пользователи (B2C)

- Пользователи, которым нужно быстро конвертировать или сжать 1–10 файлов (фото, видео, документы, аудио, архивы, электронные книги) без регистрации и без передачи файлов в облако.  
- Основные сценарии: подготовка файлов для отправки по почте/мессенджерам, загрузки на сайты и в соцсети, печати, обучения и повседневной работы.

#### 1.2.2 Профессиональные пользователи (PRO)

- Фотографы и студии: работа с RAW (CR2, NEF, ARW и др.) и HEIC/HEIF, пакетная конвертация в JPEG/PNG/TIFF с контролем качества и размера, профили «web/print/archive».  
- Дизайнеры, полиграфия, рекламные агентства: AI/EPS/CDR/SVG для логотипов, фирстиля и макетов; конвертация в PNG/PDF/SVG для печати, веба и плоттеров без установки Illustrator/CorelDRAW.  
- Издательства, self‑publishing‑авторы, образовательные проекты, библиотеки: EPUB/MOBI/AZW3/FB2 ↔ EPUB/PDF для разных ридеров и платформ.  
- 3D‑моделирование и 3D‑печать: STL/OBJ/GLTF/GLB/BLEND (с перспективой 3MF/STEP) для подготовки моделей к печати и визуализации, конвертация между полигональными и сценическими форматами.  
- Инженеры, конструкторы, архитекторы: CAD‑чертежи DWG/DXF/STEP/IGES → PDF/PNG/SVG для просмотра и согласования без AutoCAD/коммерческих САПР.  
- Разработчики и дата‑инженеры: исходный код (JS/TS, HTML, CSS), данные (JSON, Markdown, YAML), базы данных (SQL dump, SQLite) и пакеты приложений (APK, DEB, RPM, MSI) — быстрая конвертация, минификация, форматирование (pretty/minify), генерация превью и документации.

#### 1.2.3 Команды и организации (B2B, self‑hosted / on‑premise)

- Малый и средний бизнес, медиа‑ и креативные агентства, образовательные организации, компании с повышенными требованиями к безопасности (банки, госструктуры, корпоративные архивы).  
- Сценарии: развёртывание FormatHub внутри периметра, интеграция с внутренними порталами и CRM через REST API, централизованное управление лимитами и историей, соответствие корпоративным политикам приватности и регуляторики.

### 1.3 Основные сценарии использования

#### 1.3.1 Базовые сценарии

- Конвертация изображений (JPEG, PNG, WebP, GIF, BMP, TIFF, HEIC, RAW и др.) и видео (MP4, AVI, MOV, MKV, WEBM) в популярные форматы для веба, соцсетей и мессенджеров.  
- Конвертация офисных документов (DOCX, ODT, RTF, TXT, XLSX, CSV, ODS, PPTX, ODP) в PDF и обратно, где это технически возможно.  
- Сжатие изображений, видео, PDF и архивов для уменьшения размера без заметной потери качества (с управляемыми профилями качества).  
- Простые PDF‑операции: merge, split, rotate, protect, unlock, crop, extract‑images, flatten для бытовых и рабочих задач.

#### 1.3.2 PRO‑сценарии

- Пакетная конвертация RAW/HEIC → JPEG/PNG/TIFF с профилями «для печати», «для веба», «для архива».  
- Конвертация AI/EPS/CDR/SVG → PNG/PDF/SVG для типографий, рекламных агентств и производителей сувенирной продукции без установки тяжёлых редакторов.  
- Конвертация EPUB/MOBI/AZW3/FB2 ↔ EPUB/PDF для подготовки релизов книг и учебных материалов под разные магазины и ридеры.  
- 3D‑сценарии: конвертация и подготовка STL/OBJ/GLTF/GLB/BLEND (масштабирование, выравнивание, оптимизация сетки, рендер превью) для передачи в слайсеры и системы визуализации.  
- CAD‑сценарии: DWG/DXF/STEP → PDF/PNG/SVG для просмотра и печати чертежей без установки коммерческих САПР.  
- Code/Data‑сценарии: JSON pretty/minify, Markdown → HTML/PDF, HTML → PDF, JS/TS ↔ форматирование и минификация, YAML ↔ JSON.  
- Базы данных: конвертация SQL dump ↔ SQLite, экспорт таблиц в CSV/JSON.  
- Образы дисков и пакеты: распаковка и просмотр содержимого ISO/IMG/DMG, APK/DEB/RPM/MSI без установки (извлечение ресурсов, манифестов).

### 1.4 Конкурентные преимущества (анализ рынка)

Сравнение с типичными облачными сервисами конвертации файлов (FreeConvert, Convertio, FileConvertFree) актуально на июнь 2026 года.

| Функция | FormatHub | FreeConvert.com | Convertio.co | FileConvertFree.com |
|---------|-----------|-----------------|--------------|---------------------|
| Обработка файлов | Локально (офлайн, self‑hosted) | В облаке | В облаке | В облаке |
| Макс. размер файла (free) | Безлимит (тех. лимит 2 ГБ+ через env) | Около 1 ГБ (free), далее платно | Около 100 МБ (free) | Около 500 МБ (free) |
| Приватность данных | Файлы не покидают устройство/сервер | SSL + автоудаление через N часов | SSL + автоудаление | Автоудаление через несколько минут |
| Объединение файлов (merge) | PDF Tools (локально) | Да (часто Premium) | Частично / не акцентировано | Ограниченно |
| Редактор изображений | Обрезка, фильтры, качество | Базовые настройки | Минимально / нет | Есть (AI‑фичи: remove BG, upscale) |
| Сжатие (Compress) | Отдельная вкладка/режим | Да | Ограниченно | Да |
| Пакетная обработка | До 50 файлов локально | Да (часто в Premium) | Ограниченно | Ограниченно |
| API для разработчиков | В планах (self‑hosted / B2B) | Да (платный) | Да (платный) | Нет |

Уникальные преимущества FormatHub:

- Абсолютная приватность и контроль: файлы не покидают компьютер пользователя или инфраструктуру организации; нет зависимости от стороннего облака.  
- Работа без интернета: локальный режим не требует доступа к сети и подходит для закрытых контуров и офлайн‑сценариев.  
- Отсутствие аплоада: задержка связана только с реальной обработкой файла, без загрузки/выгрузки в облако.  
- Единый инструмент: конвертация, сжатие и базовое редактирование (PDF‑утилиты, обработка изображений/видео, архивы, e‑books, OCR) в одном интерфейсе.  
- Бесплатная пакетная обработка: до 50 файлов в batch без доплат и скрытых ограничений; лимиты прозрачно конфигурируются.  
- Фокус на PRO‑форматы и отрасли: RAW/HEIC, AI/EPS/CDR, EPUB/MOBI/AZW3/FB2; 3D‑модели (STL/OBJ/GLTF/GLB/BLEND), CAD (DWG/DXF/STEP), код и данные (JS/TS/HTML/CSS/JSON/MD/YAML), базы данных (SQL/SQLite), образы дисков (ISO/IMG/DMG) и пакеты приложений (APK/DEB/RPM/MSI). Базовые форматы входят в основной объём поддержки, тяжёлые и редные распределены по расширенным этапам Roadmap.

### 1.5 Принципы продукта и UX

- Минимализм и отсутствие мусора.  
  Никакой рекламы, навязчивых попапов и баннеров. Интерфейс строится вокруг основных разделов: Convert, Compress, Tools, History, Formats, How it works, About.

- Честность и предсказуемость.  
  Все ограничения, поддерживаемые форматы и системные требования явно задокументированы. Недоступные операции скрываются или помечаются на основе `GET /api/system/capabilities`, при попытке запуска возвращается понятное сообщение об ошибке.

- Доступность и производительность.  
  Поддержка клавиатурной навигации, aria‑атрибутов, корректная работа на экранах от 320 px, режим `lowPerformance` для слабых устройств, снижение визуальных эффектов при низком FPS. Анимации и эффекты не должны мешать основному сценарию.

- Расширяемость и модульность.  
  Новые форматы, инструменты, AI‑функции и интеграции добавляются через расширение конфигурации форматов (`formats.json`), модульные «обработчики» и UI‑компоненты без ломки существующих сценариев и API.

## 2. ТЕХНОЛОГИЧЕСКИЙ СТЕК

### 2.1 Фронтенд

- React 20+  
  Используются современные возможности (`use`, `useTransition`, `Suspense`) для оптимизации рендеринга, плавных переходов состояний и работы с асинхронными операциями (загрузка форматов, истории, статуса задач). React Compiler может использоваться только в режиме разработки; в production-сборке он отключён.

- Next.js 18 (App Router)  
  Унифицированный роутинг, серверные и клиентские компоненты. Основные маршруты:
  - `/convert`, `/compress`, `/tools`, `/history`, `/formats`, `/how-it-works`, `/about`;
  - поддержка префиксов локали (`/ru/...`, `/en/...`).
  Для production-сборки используется стабильный `next build` на базе webpack; Turbopack допускается только для `next dev`.

- TypeScript 6.0+  
  Строгая типизация всех модулей, общие типы для фронтенда и бэкенда (форматы, задачи, статус прогресса, capabilities). Типы задач и форматов не зависят от конкретных реализаций конвертеров.

- TailwindCSS 4  
  Ускоренная разработка интерфейса, консистентная дизайн-система (отступы, размеры шрифтов, цвета, состояния hover/focus). Используется как основа дизайн-токенов (цветовые схемы для категорий форматов, статусов задач и т.д.).

- Zustand  
  Лёгкое глобальное состояние для:
  - выбранной темы (светлая/тёмная, режим авто),
  - текущих задач и прогресса (отображение в UI),
  - настроек пользователя (язык, режим производительности, включение/отключение анимаций),
  - данных истории (кэш, фильтры, выбранный тип операций).

- Axios  
  Для REST-запросов к локальному бэкенду:
  - загрузка файлов (`/api/upload`, `/api/batch`),
  - операции сжатия и инструментов (`/api/compress`, `/api/tools/...`),
  - работа с историей (`/api/history`),
  - получение системных возможностей (`/api/system/capabilities`).
  В случае ошибок клиент ожидает структурированный JSON-ответ с кодом и локализуемым сообщением.

- Framer Motion 12  
  Плавные анимации UI:
  - появления карточек форматов, панели прогресса, модальных окон,
  - micro-interactions (hover, tap, drag),
  - базовый эффект glitch для ошибок (упрощённый, без тяжёлых шейдеров).
  При `lowPerformance = true` сложные анимации и эффект "финишной полосы" отключаются.

- `next-themes`  
  Переключение тёмной/светлой темы:
  - хранение выбора в `localStorage`,
  - учёт системной темы (`prefers-color-scheme`) при первом запуске,
  - корректная работа с SSR (без мерцания темы).

- `next-intl`  
  Интернационализация интерфейса:
  - языки: русский, английский,
  - структура переводов в JSON-файлах, ключи — стабильные идентификаторы,
  - выбор языка по URL-префиксу (`/ru`, `/en`), с резервом по системному языку,
  - fallback на `en`, логирование отсутствующих ключей для последующего заполнения.

- Lucide React  
  Набор иконок для навигации, кнопок действий и статусов. Для чисто декоративных иконок используются `aria-hidden="true"`.

- Кастомные SVG-иконки форматов  
  - Хранятся в `public/icons`,
  - используются в компонентах FormatCard, списках форматов, истории,
  - создаются/редактируются в Figma, набор пополняется по мере добавления форматов.

- Редактор изображений (UI-уровень)  
  - Реализуется как отдельный компонент с React-обёртками над Canvas API,
  - функции: обрезка, поворот, отражение, изменение размера, базовые фильтры яркости/контраста/насыщенности,
  - итоговый результат передаётся на бэкенд для финальной обработки/сохранения (через Sharp).

- Редактор видео (UI-уровень)  
  - Компонент таймлайна: выбор интервала по времени (trim),
  - отображение предпросмотра, базовые элементы управления (play/pause, scrub),
  - настройки для watermark и извлечения аудио (видимые параметры, но сама обработка выполняется на бэкенде через FFmpeg).

- React Compiler  
  - Используется только в режиме разработки для оптимизации рендеринга,
  - в production-сборке полностью отключён (флаг `compiler: false` в `next.config.js`), чтобы избежать нестабильности.

- Turbopack  
  - Применяется исключительно для `next dev` (быстрый HMR),
  - не используется для production-сборки; все прод-артефакты собираются через стабильный `next build` на webpack.

### 2.2 Бэкенд (локальный сервер)

- Node.js 22 LTS  
  Актуальная LTS-версия с поддержкой `worker_threads`, современных ES-возможностей и стабильных нативных модулей (Sharp, libvips и др.). Минимально рекомендуемая версия задаётся явно в `engines` (`package.json`).

- Express 5  
  - HTTP-сервер на порту `3001`,
  - REST-API для загрузки, обработки, истории, системных возможностей,
  - структурированные ответы в формате `{ code, message, details? }` для всех ошибок.

- Multer  
  - Приём файлов через `multipart/form-data`,
  - ограничение размера файла по умолчанию 2 ГБ (переопределяется через `MAX_FILE_SIZE_BYTES` в `.env`),
  - до сохранения выполняется проверка свободного места на диске (через `check-disk-space` или аналог); если свободно менее 500 МБ — запрос отклоняется с ошибкой `INSUFFICIENT_DISK_SPACE`.

- WebSocketServer  
  - Работает поверх того же HTTP-сервера (`ws://localhost:3001`),
  - каналы для подписки клиентов на прогресс задач и батчей,
  - формат сообщений: текущий статус (queued, running, completed, failed, canceled), прогресс (0–100), скорость, ETA, код ошибки (если есть), флаг отмены.

#### 2.2.1 Конвертеры

Выбор стратегии зависит от типа файла, целевого формата и наличия системных зависимостей.

- Изображения (JPEG, PNG, WebP, GIF, BMP, TIFF, HEIC, RAW и др.)  
  - Библиотека Sharp (нативный модуль),
  - при простых операциях (смена формата без сложной обработки) Sharp может не отдавать промежуточные события; в этом случае прогресс эмулируется (плавный рост 0–100 % с учётом размера файла и этапов),
  - для HEIC/RAW используются `libvips-plugins`; при отсутствии — ошибка `MISSING_LIBVIPS_PLUGINS` и инструкция по установке.

- Аудио/Видео (MP4, AVI, MOV, MKV, WEBM, MP3, WAV, FLAC, AAC, M4A и др.)  
  - Приоритет — системный FFmpeg (binary в PATH или типичных путях),
  - для расчёта прогресса парсится stderr FFmpeg (тайминг/кадры),
  - при отсутствии FFmpeg возвращается понятная ошибка `MISSING_FFMPEG` с инструкциями по установке под Windows/macOS/Linux,
  - `ffmpeg.wasm` в MVP не используется, чтобы не увеличивать размер фронтенда и не ухудшать производительность.

- Документы  
  - DOCX → PDF: `mammoth.js` (HTML из DOCX) + `puppeteer` (рендеринг HTML в PDF через один переиспользуемый инстанс браузера, новая страница на задачу),
  - TXT/MD → PDF: кастомный HTML-рендерер + `puppeteer` (Markdown предварительно преобразуется в HTML через `marked`/`remark`),
  - ODT/RTF → PDF и другие офисные форматы: только через установленный LibreOffice в headless-режиме,
  - при отсутствии LibreOffice — ошибка `MISSING_LIBREOFFICE` и модальное окно с инструкциями.

- Таблицы (XLSX, CSV, ODS)  
  - Используется `sheetjs` / `xlsx`,
  - при размере файла < 10 МБ обработка выполняется в основном потоке,
  - при размере ≥ 10 МБ — обработка выносится в `worker_threads`, чтобы не блокировать event loop; прогресс эмулируется по этапам (чтение, обработка, запись).

- Презентации (PPTX, ODP)  
  - Конвертация через LibreOffice (headless),
  - целевые форматы: PDF и ODP,
  - при отсутствии LibreOffice — `MISSING_LIBREOFFICE` и инструкция в UI.

- 3D‑модели (STL, OBJ, GLTF/GLB, BLEND, FBX, 3MF, PLY)  
  - Конвертация и оптимизация через Blender в headless‑режиме (`blender --background --python <script>`),
  - для лёгких сценариев (STL/OBJ/GLTF ↔ без сложных сцен) допускается использование чистых JS‑библиотек (`three.js` + загрузчики/экспортёры) без внешних зависимостей,
  - рендер превью — через Blender (Cycles/Eevee) или glTF‑рендерер в `puppeteer`,
  - при отсутствии Blender и невозможности использовать JS‑путь — ошибка `MISSING_BLENDER` и инструкция.

- CAD (DWG, DXF, STEP, IGES)  
  - Конвертация через ODA File Converter / LibreCAD / FreeCAD / Inkscape (по приоритету),
  - целевые форматы: DXF, PDF, PNG, SVG,
  - при отсутствии любого CAD‑конвертера — ошибка `MISSING_CAD_CONVERTER` и инструкция.

- Код и данные (JSON, YAML, XML, HTML, CSS, JS/TS)  
  - Pretty/minify, валидация — встроенные JS‑парсеры (`JSON.parse`, `js-yaml`, `fast-xml-parser`),
  - форматирование и минификация JS/TS/CSS/HTML — `prettier` + `terser`/`clean-css`/`html-minifier-terser`,
  - Markdown → HTML/PDF — `marked`/`remark` + `puppeteer`,
  - HTML → PDF — `puppeteer`,
  - внешние зависимости не требуются; обработка выполняется в `worker_threads` для файлов > 5 МБ.

- Базы данных (SQL dump, SQLite)  
  - SQL ↔ SQLite: `sql.js` (WASM‑сборка SQLite) или системный `sqlite3`,
  - экспорт таблиц в CSV/JSON — через `sql.js`/`better-sqlite3`,
  - просмотр схемы — парсинг `CREATE TABLE` из дампа или `sqlite_master`.

- Образы дисков (ISO, IMG, DMG)  
  - Распаковка и просмотр — 7-Zip (`7z`/`7zz`) или bsdtar (libarchive),
  - выход: ZIP содержимого образа или выборочных файлов,
  - при отсутствии и 7-Zip, и bsdtar — ошибка `MISSING_ARCHIVE_TOOL` и инструкция.

- Пакеты приложений (APK, DEB, RPM, MSI)  
  - Распаковка — 7-Zip/bsdtar (APK/MSI как архивы, DEB/RPM через ar + tar),
  - извлечение манифестов, ресурсов и иконок,
  - выход: ZIP ресурсов или отдельные выбранные файлы.

#### 2.2.2 Компрессоры (сжатие)

#### 2.2.7 Эмуляция прогресса

Не все обработчики отдают промежуточные события прогресса. Для таких операций применяется контролируемая эмуляция.

**Классификация операций по источнику прогресса:**

| Источник прогресса | Операции | Поведение |
|---|---|---|
| Нативный (парсинг stderr/stdout) | FFmpeg (видео/аудио/trim/GIF), LibreOffice (пакетный режим) | Парсинг `time=` / `frame=` из stderr FFmpeg; процент рассчитывается как `(текущее время / длительность) × 100`. Для LibreOffice — прогресс эмулируется (нет stdout-прогресса). |
| Нативный (обратные вызовы этапов) | puppeteer (DOCX→PDF: кол-во страниц), sheetjs (worker_threads: чтение/обработка/запись) | puppeteer: процент = `(страница / всего_страниц) × 100`. sheetjs: три этапа по ~33 % каждый (чтение, обработка, запись). |
| Эмуляция по размеру файла | Sharp (изображения), 3D (Blender headless), CAD‑конвертер | Линейная эмуляция: `(время_от_начала / ожидаемое_время) × 100`. Ожидаемое время рассчитывается как `baseTime × (fileSize / referenceSize)` с коэффициентами из конфигурации. |
| Эмуляция по этапам | Код и данные (prettier/terser/clean-css), базы данных (sql.js), образы дисков (7-Zip/bsdtar) | Фиксированные этапы с весами: например, «чтение» (20 %), «обработка» (50 %), «запись» (30 %). |
| Эмуляция мгновенная | Малые файлы (< 100 КБ), minify CSS/JS/JSON, валидация | Прогресс скачкообразно: 0 → 50 → 90 → 100 с интервалом 200–400 мс между шагами. |

**Алгоритм эмуляции (для операций без нативного прогресса):**

1. При старте задачи фиксируется `startTime` и `estimatedDuration` (мс).
2. `estimatedDuration` вычисляется как:
   - `baseDuration` × `sizeFactor`, где
   - `baseDuration` — базовое время для данного типа операции (из конфигурации, например, Sharp JPEG→PNG: 500 мс, Blender STL→OBJ: 3000 мс);
   - `sizeFactor` = `clamp(fileSize / referenceSize, 0.5, 5.0)` (referenceSize = 1 МБ по умолчанию).
3. Каждые `tickInterval` мс (по умолчанию 300 мс, при `lowPerformance` — 1000 мс) отправляется WebSocket‑сообщение:
   - `progress = clamp(100 × (elapsed / estimatedDuration), 0, 95)`.
   - `speed` = `(bytesProcessed? / elapsed)` или `null`.
   - `eta = max(0, estimatedDuration - elapsed)`.
4. При завершении операции прогресс принудительно устанавливается в 100 %.
5. При `lowPerformance` tickInterval увеличивается до 1000 мс; `speed` и `eta` могут не отправляться (для снижения нагрузки на рендеринг).

**Конфигурация эмуляции:**

```js
const progressConfig = {
  tickInterval: 300,          // мс между тиками WebSocket
  lowPerfTickInterval: 1000,  // мс при lowPerformance
  referenceSize: 1_048_576,  // 1 МБ
  baseDurations: {
    sharp: 500,               // мс
    puppeteer: 2000,          // мс
    sheetjs: 1000,            // мс
    blender: 3000,            // мс
    cadConverter: 4000,       // мс
    codeFormat: 300,          // мс
    database: 1500,           // мс
    archiveTool: 2000,        // мс
    sevenZip: 1500            // мс
  }
};
```

#### 2.2.8 Конкурентность и очередь задач

**Лимиты параллельных операций:**

| Параметр | Значение по умолчанию | Описание |
|---|---|---|
| `MAX_CONCURRENT_JOBS` | 2 | Максимальное количество одновременно выполняемых задач конвертации/сжатия/редактирования. |
| `MAX_CONCURRENT_BATCH_FILES` | 3 | Максимальное количество файлов, обрабатываемых параллельно внутри одного batch. |
| `MAX_BATCH_SIZE` | 50 | Максимальное количество файлов в одном batch (из раздела 1.1). |
| `MAX_WORKER_THREADS` | 4 | Максимальное количество `worker_threads` для sheetjs и code-операций. |
| `JOB_TIMEOUT_MS` | 600 000 (10 мин) | Динамический таймаут одной задачи (по умолчанию). |
| `JOB_QUEUE_MAX_PENDING` | 100 | Максимальное количество задач в очереди ожидания. |

**Поведение:**

- При поступлении нового запроса:
  - если текущее количество выполняемых задач < `MAX_CONCURRENT_JOBS` — задача запускается немедленно;
  - если лимит превышен — задача ставится в очередь со статусом `queued` и получает `jobId`.
- WebSocket для задачи в очереди:
  - отправляет начальное сообщение `{ status: "queued", position: N }`, где N — позиция в очереди;
  - при запуске — `{ status: "running", ... }`.
- Batch-задачи:
  - внутренняя очередь batch из `MAX_BATCH_SIZE` файлов обрабатывается по `MAX_CONCURRENT_BATCH_FILES` файлов параллельно;
  - прогресс batch = `(обработанные файлы / всего файлов) × 100`;
  - при ошибке одного файла batch помечается как `partial_failure`, остальные продолжают обработку; результат — ZIP с успешными файлами + отчёт об ошибках.
- Динамический таймаут:
  - `actualTimeout = JOB_TIMEOUT_MS × sizeFactor`, где `sizeFactor = clamp(fileSize / (100 МБ), 1.0, 5.0)`;
  - при превышении таймаута — задача отменяется, временные файлы удаляются, отправляется ошибка `TIMEOUT`.

**Переменные окружения:**

Все лимиты настраиваются через `.env`:
- `MAX_CONCURRENT_JOBS=2`
- `MAX_CONCURRENT_BATCH_FILES=3`
- `MAX_BATCH_SIZE=50`
- `MAX_WORKER_THREADS=4`
- `JOB_TIMEOUT_MS=600000`
- `JOB_QUEUE_MAX_PENDING=100`

#### 2.2.9 Паттерн конвертации (ConvertService)

Архитектура обработчиков строится по паттерну **стратегия (Strategy)** с единой точкой входа.

**Структура:**

```
backend/src/
├── server.ts                     # Express + WebSocket + запуск
├── routes/
│   ├── upload.ts                 # POST /api/upload
│   ├── batch.ts                  # POST /api/batch
│   ├── history.ts               # GET/DELETE /api/history
│   ├── system.ts                # GET /api/system/capabilities
│   └── download.ts               # GET /api/download/:id
├── services/
│   ├── jobQueue.ts               # очередь задач, concurrency control
│   ├── progressEmitter.ts        # WebSocket-эмуляция и нативный прогресс
│   ├── convertService.ts         # единая точка входа (фабрика стратегий)
│   └── converters/
│       ├── sharpConverter.ts     # изображения (Sharp)
│       ├── ffmpegConverter.ts    # аудио/видео (FFmpeg)
│       ├── documentConverter.ts  # DOCX/TXT/MD → PDF (mammoth + puppeteer)
│       ├── officeConverter.ts    # LibreOffice (PPTX, ODP, ODT, RTF, CAD)
│       ├── spreadsheetConverter.ts # sheetjs
│       ├── ebookConverter.ts     # pandoc / calibre
│       ├── vectorConverter.ts    # librsvg / ImageMagick
│       ├── model3dConverter.ts   # Blender headless / three.js
│       ├── cadConverter.ts       # ODA / LibreCAD / FreeCAD / Inkscape
│       ├── codeConverter.ts      # prettier / terser / clean-css / marked
│       ├── databaseConverter.ts  # sql.js / better-sqlite3
│       ├── archiveConverter.ts   # 7-Zip / bsdtar / archiver
│       ├── pdfToolsConverter.ts  # pdf-lib / Ghostscript
│       └── ocrConverter.ts       # Tesseract.js
├── utils/
│   ├── dependencyChecker.ts      # проверка системных инструментов
│   ├── diskSpace.ts              # check-disk-space
│   ├── logger.ts                 # error.log + access.log
│   └── formatRegistry.ts         # загрузка и кэширование formats.json
├── tmp/                          # временные файлы (auto-clean)
├── converted/                    # результаты (постоянные до удаления)
├── history.json                  # история операций
└── error.log                     # лог ошибок
```

**ConvertService — единая точка входа:**

- `convertService.convert(job: Job)`:
  1. Определяет тип файла (extension + MIME).
  2. Ищет запись в `formats.json` по extension/MIME.
  3. Проверяет `compatibleWith` для целевого формата.
  4. Проверяет `requires*` поля против `systemCapabilities`.
  5. Выбирает конвертер через реестр стратегий (по `category` + целевому формату).
  6. Передаёт задачу в `jobQueue`.
  7. `jobQueue` запускает конвертер при доступности слота, управляет прогрессом через `progressEmitter`.
  8. При завершении — сохраняет результат в `converted/`, записывает в `history.json`, отправляет финальный статус через WebSocket.

**Добавление нового конвертера:**

1. Создать файл `backend/src/services/converters/newTypeConverter.ts`.
2. Реализовать интерфейс `IConverter` (`convert(input: string, output: string, options: ConvertOptions): Promise<ConvertResult>`).
3. Зарегистрировать в `convertService.ts` в маппинге `category → converter`.
4. Добавить записи в `formats.json`.
5. Обновить `dependencyChecker.ts` при наличии новых системных зависимостей.

### 2.3 Инфраструктура и структура проекта

- Монорепозиторий (monorepo)  
  - Корень проекта с общими зависимостями, конфигурацией и скриптами,  
  - директории `frontend/` и `backend/`,  
  - общие типы и утилиты в отдельном пакете (например, `shared/`).

- Запуск одной командой  
  - В корне используется `concurrently` (или аналог),  
  - команда `npm run dev`:
    - запускает фронтенд (`next dev`) на порту `3000`,
    - запускает бэкенд (Express) на порту `3001`.

- Зависимости и postinstall  
  - Скрипт `postinstall`:
    - проверяет наличие FFmpeg и LibreOffice (и, при необходимости, других бинарей),
    - выводит предупреждения в консоль, если они не найдены,
    - не блокирует установку npm-зависимостей.
  - При каждом запуске сервера выполняется рантайм-проверка зависимостей и обновление `systemCapabilities`.

- Внешняя инфраструктура в MVP  
  - В MVP не используются Docker, Redis, внешние облачные сервисы, очереди сообщений и т.п., за исключением локальных системных бинарей (FFmpeg, LibreOffice и др.).
  - Предполагается запуск на локальном компьютере пользователя или на отдельном сервере в роли локального сервиса.
  - Поддержка Docker-образа, Kubernetes, внешних хранилищ (S3 и т.п.) и механизмов
обновления рассматривается как Post-MVP и отражается в Roadmap отдельными этапами.

## 3. ДИЗАЙН И ВИЗУАЛЬНАЯ ИДЕНТИЧНОСТЬ (ОСНОВНЫЕ ПРИНЦИПЫ)

### 3.1 Стиль

- Минимализм с эффектом Liquid Glass: матовое стекло, полупрозрачные панели, мягкие тени, `backdrop-blur`, аккуратные градиенты, не отвлекающие от контента.
- Две основные темы: тёмная и светлая, с сохранением выбора пользователя и учётом системной темы (`prefers-color-scheme`).
- Типографика: шрифт Inter (Google Fonts), единая шкала размеров для заголовков и текста, акцент на читаемости в интерфейсах.
- Адаптивность интерфейса от мобильных устройств (320 px) до широких экранов, без горизонтальной прокрутки основных разделов.

### 3.3 Иконки форматов

- Набор популярных форматов (25–35 штук на первом этапе) по основным категориям: изображения, видео, аудио, документы, таблицы, презентации, архивы, e-books, вектор, 3D, CAD, код и данные, базы данных, образы дисков, пакеты приложений.
- Стиль:
  - минималистичный, линейный,
  - обводка 1.5 px,
  - скруглённые углы,
  - базовый размер 32×32 px, в карточках — 48×48 px,
  - для иконок форматов используется единая база‑контейнер 40×40 px (по Figma‑макету): слой Glow (blur 4 px), слой InnerShadow (`inset 0 4px 4px 4px rgba(0,0,0,0.25)`), фон `#171717`, радиус 5 px, поверх — вектор‑глиф формата размером 16–24 px по центру.
- Процесс:
  - иконки создаются в Figma по единой сетке и гайдлайнам (E‑BOOK, PDF, DOCX, POWERPOINT, IMAGE, EXCEL, VIDEO, VECTOR, MUSIC, 3D, ZIP, JavaScript, HTML, CSS, REACT, DB, CAD и др.),
  - экспортируются в SVG и складываются в `public/icons/`,
  - в интерфейсе подключаются как inline-SVG или через отдельные React-компоненты с возможностью изменения цвета через CSS-переменные.

- Светлая тема:

- Тёмная тема:
  - поверх `#171717` и `#484848` для карточек используется `background: rgba(31, 41, 55, 0.7)`,

  - тени: `0 10px 30px rgba(0, 0, 0, 0.4)`.

#### Скругления и типографика

- Скругления:
  - карточки — 24 px,
  - кнопки — 12 px,
  - поле поиска и инпуты — 16 px,

- Типографика (Inter):
  - заголовок h1 — 32 px (weight 700),

  - disabled-элементы: осветлённые оттенки серого (например, `#A3A3A3`).

### 3.5 Фоновые градиенты и анимация завершения конвертации

#### Базовый фон

- Основной фон — однотонный (светлый или тёмный в зависимости от темы).
- Допускается использование декоративной сетки, лёгких паттернов или угловых элементов, которые постепенно растворяются (уменьшение непрозрачности, размытие) по мере приближения к центру экрана, чтобы фокус оставался на зонах взаимодействия.

#### Градиенты при выборе файлов

- Левый край (исходный формат):
  - радиальный градиент, привязанный к категории исходного формата,
  - в светлой теме: от тёмного цвета категории к светлому, затем к основному фону,
  - в тёмной теме: от светлого цвета категории к тёмному, затем к основному фону.

- Правый край (целевой формат):
  - аналогичный радиальный градиент, появляется после выбора целевого формата,
  - использует цвета категории целевого формата (светлый/тёмный).

Градиенты не должны мешать чтению текста и распознаванию элементов интерфейса; основная область работы (центральные карточки и панели) остаётся визуально нейтральной.

#### Анимация при завершении конвертации (100 % прогресса)

Анимация запускается при успешном завершении операции и длится фиксированное время (примерно 0.8–1.2 секунды):

1. По левому краю экрана появляется вертикальная полоса шириной около 10 px.  
   Цвет полосы:
   - в тёмной теме — светлый цвет исходной категории,
   - в светлой теме — тёмный цвет исходной категории.
2. Полоса плавно перемещается к правому краю экрана (CSS-анимация `transform`/`translateX`).
3. Область позади полосы (слева от неё) заполняется дополнительным цветом:
   - в тёмной теме — более тёмным оттенком категории,
   - в светлой теме — более светлым оттенком категории.
4. По мере движения полосы её цвет плавно переходит к цвету целевой категории:
   - в тёмной теме — к светлому цвету целевой категории,
   - в светлой теме — к тёмному цвету целевой категории.
5. Заливка позади полосы также постепенно трансформируется в оттенок целевой категории, создавая эффект «перехода» от исходного формата к целевому.
6. Когда полоса достигает правого края, анимация завершается, фон возвращается к базовому состоянию (исходный фон и статичные градиенты).

Реализация:

- используется CSS-анимация и/или Framer Motion, анимируются только свойства `transform` и `opacity` (без тяжёлых перерисовок);
- анимация не блокирует взаимодействие с интерфейсом.

#### Отключение анимации

- При `lowPerformance === true`:
  - анимация полосы не запускается,
  - показывается только статический градиент/статическое состояние завершения.
- При `prefers-reduced-motion: reduce`:
  - анимация также отключается или упрощается (минимальное изменение цвета без движения).
- При ошибке конвертации:
  - анимация полосы не запускается,
  - отображается glitch-эффект: кратковременное «дрожание» текста сообщения об ошибке и подсветка блока ошибки, без избыточных визуальных эффектов.

## 4. ЛОГИКА КОНВЕРТАЦИИ, СЖАТИЯ И РЕДАКТИРОВАНИЯ

### 4.1 Взаимодействие клиент–сервер (общее для всех операций)

1. Пользователь выбирает файл (drag&drop или `<input type="file">`) в одной из вкладок: **Convert**, **Compress** или **Tools**.
2. Фронтенд определяет MIME-тип, отображает иконку соответствующего формата и доступные действия. При этом:
   - используется локальный список форматов (`formats.json`);
   - дополнительно учитываются данные `GET /api/system/capabilities` (если системный инструмент недоступен, соответствующие конверсии и инструменты скрываются или помечаются как недоступные).
3. Пользователь выбирает:
   - целевой формат (вкладка Convert),
   - параметры сжатия (вкладка Compress),
   - инструменты редактирования (вкладка Tools — обрезка, объединение, защита паролем и т. д.).
4. При нажатии кнопки запуска («Конвертировать», «Сжать», «Применить»):
   - Файлы загружаются через `POST /api/upload` (FormData) с дополнительными параметрами (`operationType`, `targetFormat`, настройки качества и т. п.).
   - Для операций, требующих несколько входных файлов (объединение PDF, создание архива, batch-режим), используется `POST /api/batch` или специализированные эндпоинты (например, `/api/edit/pdf/merge`).
   - Прогресс загрузки отслеживается с помощью `axios.onUploadProgress` и отображается в интерфейсе.
5. Бэкенд создаёт задачу с `jobId` (или `batchJobId` для групповых операций), возвращает его фронтенду и запускает асинхронную обработку.
6. Фронтенд:
   - открывает WebSocket-соединение `ws://localhost:3001`;
   - отправляет сообщение подписки: `{ type: 'subscribe', jobId }` или `{ type: 'subscribe', batchJobId }`.
7. Сервер в процессе работы отправляет через WebSocket сообщения в формате:
   - `{ status: 'processing', progress, speed?, eta? }`,
   - `{ status: 'completed', progress: 100, downloadUrl }`,
   - `{ status: 'failed', error }`.
   Для пакетных операций `progress` — среднее значение по всем файлам; при необходимости могут отправляться детальные статусы по каждому файлу.
8. Отмена операции:
   - выполняется через `POST /api/cancel/:jobId` или `POST /api/cancel/:batchJobId`,
   - на фронтенде вызывается по кнопке «Отмена» в панели прогресса,
   - сервер завершает задачу, очищает временные файлы и отправляет через WebSocket `{ status: 'canceled', message: 'Operation canceled by user' }`.
9. Скачивание результата:
   - по ссылке `GET /downloads/:jobId` (один файл),
   - для batch-операций — `GET /downloads/:batchJobId` (ZIP-архив с результатами).

### 4.2 Бэкенд: управление задачами и очередью

**Хранилище задач:**

- Все активные задачи хранятся в памяти в структуре вида `Map<string, Job>`.
- При перезапуске сервера активные задачи теряются, незавершённые временные файлы очищаются — поведение допустимо для локального сценария.

**Хранилище истории:**

- История операций хранится в файле `history.json` в директории `backend/`.
- При старте сервера:
  - файл истории загружается;
  - для каждой записи проверяется существование физического файла по `downloadPath`;
  - записи с отсутствующими файлами удаляются.
- При выполнении новых операций в историю добавляются записи с:
  - `id`, `originalName`, `operation`, `targetFormat`, `sizeBytes`, `convertedAt`, `downloadPath`, `operationParams`.

### 4.4 Отмена операции (единый механизм)

- Для каждой задачи в `jobManager` хранится `AbortController`.
- Вызов `POST /api/cancel/:jobId` или `POST /api/cancel/:batchJobId`:
  - вызывает `controller.abort()` для соответствующей задачи;
  - внутри Processors обработчик `signal.aborted` выполняет:
    - остановку дочерних процессов (`child_process.kill('SIGTERM')` для FFmpeg, LibreOffice, Ghostscript, pandoc, calibre, node-7z),
    - завершение worker-потоков (`worker.terminate()` для sheetjs),
    - закрытие браузера (`browser.close()` для puppeteer),
    - удаление временных файлов из `./uploads` и `./tmp`.
- После отмены:
  - задача удаляется из `Map`,
  - через WebSocket отправляется `{ status: 'canceled', message: 'Operation canceled by user' }`,
  - фронтенд обновляет состояние панели прогресса.

### 4.5 Пакетная обработка (Batch)

- Пакетный режим принимает массив элементов вида `{ file, params }`, где `params` описывает тип операции и её настройки.
- Ограничения по умолчанию:
  - не более 50 файлов за одну batch-операцию;
  - суммарный размер файлов — не более 10 ГБ (значение настраивается через `.env`).

**Поведение:**

- Для одной batch-операции создаётся:
  - `batchJobId` — общий идентификатор,
  - внутренние подзадачи для каждого файла (могут иметь свои `jobId`).
- Обработка файлов выполняется последовательно (по умолчанию) для минимизации нагрузки на систему; в будущем допускается ограниченный параллелизм (например, по 2–3 задачи).
- Общий прогресс рассчитывается как среднее по всем файлам:
  - `batchProgress = sum(progress_i) / N`.
- После успешной обработки всех файлов:
  - создаётся ZIP-архив в директории `./converted/batch_<batchJobId>/`,
  - формируется URL для скачивания архива, который отправляется через WebSocket (`downloadUrl`).

**Отмена batch:**

- Вызов `POST /api/cancel/:batchJobId`:
  - прерывает текущий обрабатываемый файл,
  - отменяет все оставшиеся в очереди,
  - удаляет уже подготовленные результаты и временные файлы,
  - не создаёт архив.
- Пользователь получает статус `canceled` по WebSocket; фронтенд отображает отмену всей batch-операции.

## 5. ДЕТАЛЬНОЕ ОПИСАНИЕ ИНТЕРФЕЙСА

### 5.1 Глобальная структура и навигация

Приложение имеет единую шапку (App Bar) для всех страниц, оформленную в стиле Supabase / Resend / Vercel: логотип слева, навигация по центру, компактные переключатели справа. На мобильных устройствах шапка трансформируется в компактную версию с выезжающим меню.

#### 5.1.1 Верхняя панель (App Bar)

**Расположение и состав (десктоп):**

- **Слева — логотип:** иконка + текст «FormatHub». Клик по логотипу открывает главную страницу (`/`).
- **По центру — навигация:** 5 пунктов-иконок. Каждый пункт в покое — только иконка (статичная, без подписи). При наведении курсора справа от иконки плавно появляется текстовая подпись, а снизу выпадает контекстное меню.

**Навигационные пункты (центральная группа):**

| # | Иконка | Подпись (tooltip/hover) | Назначение | Маршрут |
|---|--------|--------------------------|------------|---------|
| 1 | 🏠 Дом | «Home» / «Главная» | Главная страница (лендинг, презентация поддерживаемых форматов) | `/` |
| 2 | 📄 Файл (со стрелками) | «Convert» / «Конвертация» | Конвертация файлов | `/convert` |
| 3 | ✏️ Карандаш | «Tools» / «Инструменты» | Редактирование (изображения, видео, PDF, архивация, OCR, batch) | `/tools` |
| 4 | ⏳ (TBD) | «History» / «История» *(вариант)* | Список всех конвертаций пользователя | `/history` |
| 5 | ℹ️ (TBD) | «About» / «О проекте» *(вариант)* | Сведения о проекте / профиль | `/about` |

> **Назначение пунктов 4 и 5 пока не финализировано.** Кандидаты: «History» (история конвертаций), «About» (о проекте), «Profile» (профиль пользователя), «Compress» (сжатие, если решено вынести в отдельную вкладку). Иконки для пунктов 4 и 5 подбираются после утверждения назначения.

**Поведение навигационного пункта при наведении (как у Resend):**

1. **В покое:** отображается только иконка в круглой/скруглённой «пилюле». Цвет иконки — `var(--secondary-white)` (приглушённый), фон пилюли прозрачный или слегка подсвеченный.
2. **Наведение курсора:**
   - справа от иконки **плавно (быстро, ~150–200 мс)** выезжает текст подписи;
   - пилюля растягивается по ширине (иконка + текст), ширина = содержимому;
   - фон пилюли меняется на `var(--secondary-black)` с лёгкой подсветкой.
3. **После раскрытия подписи** снизу (под пилюлей) **выпадает меню:**
   - ширина меню = ширине раскрытой пилюли (иконка + текст);
   - длина (высота) — без жёстких ограничений, но желательно компактная (1–5 пунктов);
   - содержит под-пункты или быстрые ссылки раздела (например, для «Tools»: «Изображения», «Видео», «PDF», «Архивы», «OCR», «Batch»);
   - анимация появления — fade + slide-down, ~150 мс.
4. **Уход курсора:** меню и подпись плавно сворачиваются в обратном порядке.

**Справа — переключатели:**

- Переключатель темы (иконка Солнце/Луна) — мгновенная смена темы без перезагрузки страницы.
- Переключатель языка (кнопка «RU»/«EN») — смена локали, редирект на ту же страницу с префиксом `/ru` или `/en`.

**Адаптивность (мобильные, ширина < 768px):**

- навигация сворачивается в гамбургер‑меню (иконка слева или справа от логотипа);
- при открытии гамбургера — вертикальное меню с иконкой + текстом для каждого пункта (без hover-раскрытия);
- переключатели темы и языка остаются видимыми в App Bar.

**Доступность:**

- каждый навигационный пункт доступен с клавиатуры (`Tab`), имеет `aria-label`;
- выпадающее меню открывается по `Enter`/`Space` и закрывается по `Esc`;
- активный пункт (текущая страница) подсвечивается градиентом своей категории (см. 3.2) или акцентным цветом.

#### 5.1.2 Фоновые градиенты и анимация завершения

- Базовый фон:
  - светлая тема — однотонный фон `#E1E1E1` (или близкий к нему), с возможной декоративной сеткой/угловыми элементами, растворяющимися к центру;
  - тёмная тема — однотонный фон `#171717` (или `#111827`), также с мягкими декоративными элементами, не отвлекающими от контента.
- Градиенты при выборе файлов/действий:
  - при выборе исходного файла слева активируется радиальный градиент категории исходного формата:
    - в светлой теме: от тёмного цвета категории → к светлому → к базовому фону;
    - в тёмной теме: от светлого цвета категории → к тёмному → к базовому фону;
  - при выборе целевого формата (Convert) или действия (Compress/Tools) аналогичный радиальный градиент появляется справа, с цветами категории целевого формата.

**Анимация при завершении операции (100 %):**

1. По левому краю экрана появляется вертикальная полоса шириной ≈ 10px.
   - В тёмной теме — цвет полосы = светлый цвет категории исходного формата.
   - В светлой теме — цвет полосы = тёмный цвет категории исходного формата.
2. Полоса плавно движется к правому краю экрана.
3. Позади полосы (слева от неё) экран постепенно заливается дополнительным цветом:
   - в тёмной теме — более тёмный оттенок исходной категории;
   - в светлой теме — более светлый оттенок исходной категории.
4. По мере движения полосы её цвет плавно переходит к цвету целевой категории:
   - в тёмной теме — к светлому цвету категории целевого формата;
   - в светлой теме — к тёмному цвету категории целевого формата.
5. Заливка позади полосы также трансформируется в оттенок целевой категории, создавая визуальный переход «исходный формат → целевой формат».
6. При достижении правого края полоса исчезает, фон возвращается к базовому состоянию, остаются только статичные градиенты по краям.

**Отключение анимации:**

- При `lowPerformance === true`:
  - анимация полосы полностью отключается;
  - используются только статичные градиенты по краям без движения.
- При ошибке операции анимация полосы не запускается — сразу показывается glitch‑эффект ошибки.

#### 5.1.3 Главная страница (`/`) и карточки форматов

Главная страница — это лендинг в стиле Supabase / Vercel: крупный hero‑блок с названием и кратким описанием, ниже — **презентация поддерживаемых форматов** в виде горизонтально прокручиваемой (или сеткой) ленты карточек. Каждая карточка = один формат или одна категория.

**Карточка формата — визуальная модель:**

Карточка имеет два состояния — **покой** и **наведение (hover)**. Цвета берутся из CSS‑переменных (см. раздел 3.4 и `globals.css`):

| Параметр | Покой | Наведение (hover) |
|----------|-------|-------------------|
| Фон карточки (тёмная тема) | `var(--primary-black)` (`#171717`) | `var(--primary-black)` (`#171717`) — без изменений |
| Фон карточки (светлая тема) | `var(--primary-white)` (`#D1D1D1`) | `var(--primary-white)` (`#D1D1D1`) — без изменений |
| Обводка (border) | `var(--border)` (`#636363`) — нейтральная | **плавно** переходит в `conic-gradient` категории (из 3.2) |
| Свечение (подслой `::before`, `filter: blur`) | отсутствует (`blur(0)` / `opacity: 0`) | **плавно** появляется: `blur` увеличивается, `opacity → 1`, цвет = `glow` категории |
| Иконка/текст внутри | цвет `var(--secondary-white)` | цвет `var(--primary-white)` |

**Анимация перехода (покой → hover):**

- Длительность: **короткая и быстрая**, ~200–250 мс.
- Функция времени: `cubic-bezier(0.4, 0, 0.2, 1)` (ease-out) — «плавно, но быстро».
- Анимируются одновременно (синхронно, одна длительность):
  - `border-color` / `border-image` — нейтральный `--border` → градиент категории;
  - `opacity` подслоя свечения — `0` → `1`;
  - `filter: blur()` подслоя — `blur(0px)` → `blur(8px)` (усиление свечения);
  - (опционально) лёгкий `transform: translateY(-2px)` для подъёма карточки.
- Обратный переход (hover → покой) — те же параметры в обратную сторону, та же длительность.

**Реализация подслоя свечения:**

- Карточка — `position: relative; overflow: visible`.
- Подслой `::before`:
  - `position: absolute; inset: -8px;` (выходит за границы карточки на 8px — формирует halo);
  - `background: var(--card-glow)` (доминирующий цвет категории);
  - `border-radius: inherit + 8px`;
  - `filter: blur(8px)` в состоянии hover, `blur(0)` в покое;
  - `opacity: 0` в покое, `1` в hover;
  - `z-index: -1`.
- Градиентная обводка реализуется через `border-image` или `background-clip: padding-box/border-box` с двойным фоном.

**CSS‑переменные карточки (задаются через инлайн‑стиль или data-атрибут):**

```css
.format-card {
  --card-gradient: <conic-gradient категории>;  /* из таблицы 3.2 */
  --card-glow:     <цвет glow категории>;       /* из таблицы 3.2 */
}
```

**Поведение в разных контекстах:**

- **Главная страница (лента/сетка презентации):** карточки показывают базовую анимацию покой↔hover, как описано выше. Это «демо» поддерживаемых форматов.
- **Страница `/formats`:** карточки также используют hover‑анимацию, плюс кликабельны (открывают детали формата).
- **Выбор целевого формата (Convert):** карточка выбранного формата фиксируется в «активном» состоянии (постоянный градиент + свечение), а не только на hover.

**Режим низкой производительности (`lowPerformance`):**

- Подслой свечения полностью отключается (`::before { display: none }`).
- Переход обводки `--border` → градиент остаётся, но без аcolourного halo (только `border-color`/`border-image`, без blur).
- `transform` (подъём карточки) отключается.

**Доступность:**

- Карточка доступна с клавиатуры; при `:focus-visible` применяется тот же эффект, что и при `:hover`.
- Информация о категории формата дублируется в `aria-label` и/или `data-category` (не только цветом).

### 5.2 Вкладка «Convert» (конвертация)

#### 5.2.1 Зона загрузки файла (FileUploader)

- Расположение: центр экрана; на десктопе ширина до 600px, высота 280px.
- Оформление: Liquid Glass (полупрозрачный фон, `backdrop-blur`, мягкая тень, скругление 24px).
- Пакетный режим:
  - чекбокс «Пакетная обработка» рядом с зоной;
  - при включении зона принимает несколько файлов (до 50);
  - под зоной отображается список файлов с иконкой, именем, размером и кнопкой удаления.

**Поддерживаемые форматы (в едином тексте):**

- Изображения: JPEG, PNG, WebP, GIF, BMP, TIFF, HEIC/HEIF, RAW (CR2, NEF, ARW и др.), AVIF.
- Видео: MP4, AVI, MOV, MKV, WEBM.
- Аудио: MP3, WAV, OGG, FLAC, AAC, M4A, WMA.
- Документы: PDF, DOCX, ODT, RTF, TXT, MD.
- Таблицы: XLSX, CSV, ODS.
- Презентации: PPTX, ODP.
- Архивы: ZIP, RAR, 7z, TAR, GZ.
- Электронные книги: EPUB, MOBI, AZW3, FB2.
- Векторная графика: SVG, AI, EPS, CDR.
- 3D‑модели: STL, OBJ, GLTF/GLB, BLEND, FBX, 3MF.
- CAD: DWG, DXF, STEP, IGES.
- Код и данные: JavaScript/TypeScript, HTML, CSS, JSON, YAML, XML.
- Базы данных: SQL dump, SQLite.
- Образы дисков: ISO, IMG, DMG.
- Пакеты приложений: APK, DEB, RPM, MSI.

Наличие некоторых операций для форматов зависит от системных инструментов (FFmpeg, LibreOffice, Ghostscript, pandoc, calibre и т. д.) и учитывается при фильтрации целевых форматов.

**Состояния (одиночный файл):**

1. Ожидание файла
   - Пунктирная рамка нейтрального серого цвета.
   - Иконка «Cloud Upload» (48px).
   - Текст: «Перетащите файл сюда или нажмите для выбора».
   - Подтекст: «Максимальный размер: 2 ГБ (можно изменить в настройках). Поддерживаются изображения, видео, аудио, документы, таблицы, презентации, архивы, электронные книги, векторная графика, 3D, CAD, код и данные, базы данных, образы дисков, пакеты приложений».
   - Кнопка «Выбрать файл» (primary).

2. Наведение (drag over)
   - Рамка становится сплошной, цвет — акцентный (синий/категорийный).
   - Фон слегка изменяется (подсветка).
   - Текст: «Отпустите файл для загрузки».

3. Файл выбран
   - Высота блока уменьшается до ~160px.
   - Отображаются:
     - иконка формата (SVG, 48×48);
     - имя файла (с обрезкой и тултипом);
     - размер (КБ/МБ/ГБ);
     - крестик для сброса;
     - индикатор статуса:
       - зелёная галочка — формат поддерживается хотя бы одним конвертером;
       - жёлтый/красный знак «!» — при проблеме, с тултипом «Формат не поддерживается ни одним конвертером».

4. Ошибка загрузки (неверный тип, превышен лимит, мало места)
   - Рамка красного цвета.
   - Однократный glitch + лёгкая тряска зоны.
   - Красный текст ошибки под блоком.
   - Кнопка «Попробовать снова».

**Состояния (пакетный режим):**

- Drag & Drop пакетной загрузки:
  - при перетаскивании нескольких файлов (или папки) — зона принимает все поддерживаемые файлы, неподдерживаемые подсвечиваются красным и отсеиваются с тостом «N файлов пропущено: неподдерживаемый формат»;
  - дубликаты по имени+размеру отсеиваются с предупреждением;
  - при превышении лимита в 50 файлов — тост «Максимум 50 файлов. Первые 50 добавлены, остальные проигнорированы».
- Список файлов:
  - каждая строка: иконка, имя, размер, статус (✓, !), кнопка удаления;
  - сверху — кнопка «Добавить ещё файлы» (открывает системный диалог, мультивыбор);
  - сортировка: по порядку добавления (по умолчанию); опционально — по имени/размеру через выпадающий селектор.
- Для каждого файла:
  - отдельный выбор целевого формата (dropdown справа от строки),
  - кнопка «Единый формат для всех» — при клике открывает модальное окно с выбором одного целевого формата, применяется ко всем файлам;
  - файлы с одинаковым исходным форматом получают одинаковый целевой формат по умолчанию (если он совместим).
- Валидация перед запуском:
  - при наличии хотя бы одного файла без выбранного целевого формата — кнопка «Конвертировать» disabled, под списком предупреждение «Выберите целевой формат для всех файлов»;
  - при наличии файла с неподдерживаемой конвертацией — строка подсвечивается красным, кнопка «Конвертировать» disabled, текст «Некоторые файлы не могут быть конвертированы в выбранный формат».

#### 5.2.2 Информация о выбранном файле

- Расположение: сразу под зоной загрузки (в одиночном режиме).
- Содержит:
  - иконку формата (32×32);
  - MIME‑тип (например, `application/pdf`);
  - размер в байтах;
  - кнопку «Изменить файл».

#### 5.2.3 Поиск целевого формата (FormatSearch)

- Поле ввода:
  - плейсхолдер: «Введите формат (например, PDF, MP3, ZIP, EPUB) или выберите из списка»;
  - иконка лупы слева;
  - кнопка очистки справа, появляющаяся при вводе;
  - debounce ввода ~200 мс.
- Умные подсказки (AI‑подобные сценарии):
  - на основе категории исходного файла и популярных комбинаций:
    - «PNG → WebP (сжать для веба)»,
    - «DOCX → PDF (сохранить макет)»,
    - «MP4 → GIF (создать анимацию)»,
    - «HEIC → JPG (совместимость с Windows)»,
    - «EPUB → PDF (читать на любом устройстве)»,
    - «ZIP → 7z (лучшее сжатие)»,
    - «SVG → PNG (растеризовать для веба)»,
    - «STL → OBJ (для 3D‑пакетов)»,
    - «GLTF → GLB (упаковать в один файл)»,
    - «DWG → PDF (посмотреть чертёж)»,
    - «JSON → YAML (читаемый конфиг)»,
    - «Markdown → PDF (документация)»,
    - «HTML → PDF (сохранить страницу)»,
    - «SQLite → CSV (выгрузить таблицу)»,
    - «ISO → ZIP (распаковать образ)»,
    - «APK → ZIP (извлечь ресурсы)».
  - отображаются вверху списка с иконкой «магическая палочка».
- Выпадающий список:
  - появляется при фокусе или вводе текста;
  - содержит только реально доступные форматы:
    - фильтрация по `formats.json` + `GET /api/system/capabilities`;
  - каждая строка:
    - иконка формата;
    - название (PDF, ZIP, EPUB);
    - расширение (`.pdf`, `.zip`, `.epub`) серым;
    - цветная метка категории.
  - если формат требует отсутствующую зависимость:
    - строка серым;
    - справа подпись «Требуется LibreOffice/FFmpeg/Ghostscript/…»;
    - выбор неактивен.
- Быстрые кнопки:
  - до 4 популярных целевых форматов для категории исходного файла;
  - пример: изображения — JPEG, PNG, WebP, GIF; архивы — ZIP, RAR, 7z; e‑books — PDF, EPUB; вектор — PNG, PDF, SVG; 3D — STL, OBJ, GLTF; CAD — PDF, PNG, DXF; код/данные — JSON, YAML, HTML; базы данных — SQL, CSV, JSON; образы дисков — ZIP, ISO; пакеты приложений — ZIP;
  - кнопки не показываются, если формат недоступен из-за отсут��твия зависимости.
- Поведение:
  - выбор формата кликом (список, подсказка, быстрая кнопка) → установка значения поля, закрытие списка;
  - при попытке выбрать очевидно некорректное соответствие (PDF → MP3) — под полем сообщение об ошибке, кнопка «Конвертировать» неактивна.

#### 5.2.4 Кнопка «Конвертировать»

- Расположение: под поиском и быстрыми кнопками.
- Состояния:
  - disabled — пока не выбран файл или целевой формат (для batch — пока не заданы форматы всем выбранным файлам);
  - active — при готовности к запуску;
  - loading — после клика (текст «Подготовка…», спиннер).
- При клике:
  - отправка `POST /api/upload` (одиночный) или `POST /api/batch` (batch) с параметрами операции;
  - вывод прогресса загрузки (в кнопке или над ней);
  - после получения `jobId`/`batchJobId` — открытие WebSocket и показ панели прогресса (5.2.5).

#### 5.2.5 Панель конвертации (ProgressPanel)

- Появляется после получения `jobId` и подписки на WebSocket.
- Расположение: под зоной загрузки; зона может схлопываться, чтобы освободить место.
- Анимация появления: выезд/фейд через Framer Motion (либо мгновенно при `lowPerformance`).

**Содержимое:**

- Иконка исходного и целевого формата (слева и справа) со стрелкой между ними (→). Для batch — иконка «несколько файлов».
- Линейный прогресс‑бар (от 0 до 100 %).
- Цифровой процент выполнения.
- Доп. информация (при наличии):
  - скорость (MB/s),
  - оценка времени до завершения (ETA),
  - статус текста («Конвертация…», «Подготовка…», «Финализация…»).
- Кнопка «Отмена» — доступна всё время до завершения.

**Состояния:**

- В процессе — прогресс обновляется по WebSocket в реальном времени.
- Успех:
  - прогресс = 100 %;
  - кнопка «Отмена» заменяется:
    - на «Скачать» (primary) — скачивание результата (`/downloads/:jobId` или ZIP для batch),
    - на «Новая конвертация» — reset состояния и возврат к исходному экрану;
  - запись добавляется в историю, файлы сохраняются в `./converted`.
- Ошибка:
  - прогресс останавливается;
  - отображается текст ошибки;
  - однократный glitch + лёгкая тряска экрана;
  - кнопки: «Закрыть» и «Попробовать снова».
- Отмена:
  - после успешной отмены: статус «Конвертация отменена», результат удалён;
  - через ~2 секунды — возврат к зоне загрузки.

#### 5.2.6 Адаптивность вкладки Convert (< 640 px)

- Зона загрузки — почти на всю ширину (отступы 16 px).
- Поиск формата — во всю ширину.
- Быстрые кнопки — сеткой по 2 в строке.
- Панель прогресса — вертикальная компоновка (иконки сверху/снизу, progress bar на всю ширину).
- Кнопки «Отмена» / «Скачать» — полноширинные.
- В batch‑режиме список файлов отображается списком «одна строка — один файл» с кнопкой удаления справа или снизу.

### 5.3 Страница истории (`/history`)

#### 5.3.1 Верхняя часть страницы

- Заголовок «История операций».
- Кнопка «Очистить всё» (красная, с подтверждением в модальном диалоге).
- Поле поиска по имени исходного файла (debounce 300 мс).
- Фильтр по типу операции:
  - выпадающий список: «Все», «Конвертация», «Сжатие», «Редактирование»;
  - фильтр применяется без перезагрузки страницы.

#### 5.3.2 Список записей (HistoryList)

Формат одной записи:

- Иконка исходного формата (32×32) или иконка типа операции (если форматов несколько).
- Имя исходного файла (обрезается, с тултипом).
- Тип операции:
  - иконка + текст («Конвертация», «Сжатие», «Редактирование»).
- Целевой формат / действие:
  - для конвертации — иконка + расширение;
  - для сжатия — текст «качество X %» или пресет;
  - для редактирования — список ключевых действий («обрезка», «фильтр» и т. п.).
- Размер результирующего файла.
- Дата операции (локализовано).
- Кнопка «Скачать» (вызов `/downloads/:fileId`).
- Кнопка «Удалить» (удаляет запись и файл).

Состояния:

- Пустая история — заглушка: «Нет завершённых операций. Перейдите на Convert, Compress или Tools, чтобы создать первую».
- Поиск/фильтр не дали результатов — «Ничего не найдено».
- При удалении — анимация исчезновения строки.
- При старте сервера битые записи (файл отсутствует) автоматически удаляются и пользователю не показываются.
- Пагинация:
  - при количестве > 20 — кнопка «Загрузить ещё» или бесконечный скролл;
  - в MVP допустимо ограничение 50 записей с пагинацией по 10.

#### 5.3.3 Адаптивность истории

- На мобильных (< 640 px) каждая запись — вертикальный блок:
  - иконка слева/сверху;
  - текстовые данные под ней;
  - кнопки «Скачать» и «Удалить» — во всю ширину (одна под другой или рядом).

  - 3D‑модели: STL, OBJ, GLTF/GLB, BLEND, FBX, 3MF.
  - CAD: DWG, DXF, STEP, IGES.

  - добавление `?format=<расширение>` к URL;
  - если на Convert уже выбран файл — формат подставляется в поле поиска и сразу активирует кнопку «Конвертировать»;
  - если файл не выбран — формат предустанавливается, пользователь видит его в поле.

**Доступность форматов:**

- Формат доступен:
  - карточка нормального вида, hover‑эффект (подъём, тень, рамка).
- Формат требует отсутствующую зависимость:
  - карточка серым;
  - текст «Требуется LibreOffice/FFmpeg/…»;
  - клик открывает модальное окно с инструкцией установки, без перехода на Convert.
- Если состояние зависимости определить нельзя — формат считается доступным, но возможна ошибка на бэкенде (обработана по общим правилам).

Адаптивность:

- Десктоп — 4–5 карточек в строке.
- Планшет — 3 карточки.
- Мобильный — 2 карточки.

#### 5.4.2 Поиск по форматам (опционально)

- Поле «Поиск формата (например, EPUB, HEIC, 7z)…» над сеткой.
- Фильтрация по названию, расширению и категории.
- Debounce 200 мс, подсветка совпадающих фрагментов.

   - OCR: распознавание текста (RU/EN).

### 5.7 Глобальные компоненты и микро‑взаимодействия

#### 5.7.1 Модальное окно отсутствующих зависимостей

- Появляется при попытке выполнить операцию, требующую недоступный системный инструмент (по данным `/api/system/capabilities`).
- Поддерживаемые проверки: FFmpeg, LibreOffice, Ghostscript, pandoc/calibre, libvips‑plugins, Tesseract.
- Структура:
  - заголовок: «Отсутствует FFmpeg» / «Отсутствует LibreOffice» и т. п.;
  - текст: короткое объяснение и инструкция установки для Windows/macOS/Linux;
  - ссылки на официальные сайты;
  - кнопки:
    - «Закрыть»;
    - «Проверить снова» (повторный запрос `/api/system/capabilities`).
  - для Tesseract.js — отдельная кнопка «Загрузить модель», если загрузка не удалась.

#### 5.7.2 Toast‑уведомления

- Используются для неблокирующих событий:
  - загрузка файла,
  - постановка задачи в очередь,
  - удаление файла из истории,
  - очистка истории,
  - успешный OCR,
  - создание/распаковка архива,
  - сохранение настроек.
- Позиция:
  - на десктопе — в правом верхнем углу;
  - на мобильных — в нижней части экрана.
- Типы:
  - success (зелёный фон, иконка галочки),
  - error (красный фон, иконка крестика),
  - info (синий фон, иконка «i»).
- Максимум 3 тоста одновременно; остальные в очереди (FIFO).

#### 5.7.3 Glitch + тряска при ошибках

- Триггер: любая критическая ошибка (HTTP 4xx/5xx, таймаут, отсутствие зависимостей, недостаточно места, ошибка OCR/архива).
- Эффекты:
  - тряска внешнего контейнера (`.shake` — короткая анимация 0.3 c);
  - glitch‑эффект текста ошибки (псевдоэлементы с RGB‑смещением).
- После анимации остаётся обычный текст ошибки.
- При `lowPerformance` тряска отключается, остаётся только лёгкий glitch текста.

#### 5.7.4 Переключение темы

- Реализация через `next-themes`:
  - темы: `light`, `dark`;
  - начальная тема — по `prefers-color-scheme` или сохранённому значению.
- Светлая тема:
  - фон: `#E1E1E1` / `#F9FAFB`;
  - карточки: `rgba(255,255,255,0.7)` + blur;
  - тени: `#0000001A`.
- Тёмная тема:
  - фон: `#171717` / `#111827`;
  - карточки: `rgba(31,41,55,0.7)` + blur.
- Выбор темы сохраняется в `localStorage`.

#### 5.7.5 Переключение языка

- Библиотека `next-intl`, локаль в URL.
- Смена языка:
  - редирект на ту же страницу с другим префиксом;
  - состояние страницы (выбранный файл) может сбрасываться;
  - опционально: сохранение выбора файла/формата в `localStorage` для восстановления.
- Локализуются все тексты, в том числе ошибки, подсказки, названия форматов, даты.

### 5.8 Поведение при низкой производительности

Механизм автоопределения:

1. При первом открытии страницы (в активной вкладке) запускается тест FPS на 2 секунды.
2. Используется `requestAnimationFrame` + простая CSS‑анимация скрытого элемента.
3. Вычисляется средний FPS; при `avgFPS < 30` или `prefers-reduced-motion: reduce` или `navigator.hardwareConcurrency < 4` → `lowPerformance = true`.
4. Повторный тест не чаще, чем раз в 10 секунд; при смене размеров окна или темы тест может запускаться заново.
5. Если вкладка неактивна, тест откладывается.

Изменения при `lowPerformance = true`:

- отключение анимаций Framer Motion;
- замена `backdrop-blur` на непрозрачный фон;
- статичные боковые градиенты;
- отключение тряски экрана (только glitch текста);
- уменьшение частоты обновления прогресса до 1 раза в секунду;
- отключение анимации движущейся полосы завершения.

Ручное управление (иконка «Настройки» в App Bar):

- Опции:
  - «Автоопределение производительности» (on/off);
  - «Отключить все анимации» (принудительный `lowPerformance = true`);
  - «Включить все анимации» (принудительный `lowPerformance = false`).
- Настройки сохраняются в `localStorage`.

## 6. ФОРМАТЫ И ДАННЫЕ (ТЕХНИЧЕСКАЯ ЧАСТЬ)

### 6.1 Хранение данных о форматах

- Файл: `frontend/public/formats.json`.
- Структура записи:

```json
{
  "name": "ZIP Archive",
  "extension": "zip",
  "category": "archives",
  "icon": "archive.svg",
  "colorLight": "#FFA000",
  "colorDark": "#C67100",
  "mimeTypes": ["application/zip", "application/x-zip-compressed"],
  "requiresLibreOffice": false,
  "requiresFfmpeg": false,
  "requiresGhostscript": false,
  "requiresPandoc": false,
  "requiresLibvipsPlugins": false,
  "requiresTesseract": false,
  "compatibleWith": ["7z", "rar", "tar", "gz"],
  "popular": true
}
```

- Поле `category` принимает значения:
  - `images` — изображения (JPEG, PNG, WebP, GIF, BMP, TIFF, HEIC, RAW, AVIF).
  - `video` — видео (MP4, AVI, MOV, MKV, WEBM).
  - `audio` — аудио (MP3, WAV, OGG, FLAC, AAC, M4A, WMA).
  - `documents` — документы (PDF, DOCX, ODT, RTF, TXT, MD).
  - `spreadsheets` — таблицы (XLSX, CSV, ODS).
  - `presentations` — презентации (PPTX, ODP).
  - `archives` — архивы (ZIP, RAR, 7z, TAR, GZ).
  - `ebooks` — электронные книги (EPUB, MOBI, AZW3, FB2).
  - `vector` — векторная графика (SVG, AI, EPS, CDR).
  - `3d` — 3D‑модели (STL, OBJ, GLTF/GLB, BLEND, FBX, 3MF).
  - `cad` — инженерная графика и САПР (DWG, DXF, STEP, IGES).
  - `code` — исходный код и данные (JS/TS, HTML, CSS, JSON, YAML, XML).
  - `database` — базы данных (SQL dump, SQLite).
  - `disk` — образы дисков (ISO, IMG, DMG).
  - `apps` — пакеты приложений и установщики (APK, DEB, RPM, MSI).
- Поля зависимостей:
  - `requiresLibreOffice`, `requiresFfmpeg`, `requiresGhostscript`, `requiresPandoc`, `requiresLibvipsPlugins`, `requiresTesseract`, `requiresBlender`, `requiresCadConverter`, `requires7zip` — используются:
    - для фильтрации доступных форматов на фронтенде;
    - для подсветки форматов, требующих установку доп. инструментов;
    - для выбора текста в модальных окнах с инструкциями.
- Поле `compatibleWith`:
  - массив расширений, в которые можно конвертировать данный формат;
  - используется для построения списка доступных целевых форматов.
- Поле `popular`:
  - флаг «популярного» формата в своей категории;
  - из популярных форматов (до 4 штук на категорию) формируются быстрые кнопки в интерфейсе.

### 6.2 Логика фильтрации целевых форматов

- При выборе исходного файла:
  - определяется расширение и MIME‑тип;
  - находиться запись(и) в `formats.json` с соответствующим `extension`/`mimeTypes`.
- Фильтрация целевых форматов:
  - берётся объединённый список всех форматов, у которых:
    - исходный формат указан в `compatibleWith`,
    - или существует явное правило конвертации (например, «любые изображения → ZIP» для архивации — задаётся в коде/доп. конфигурации).
  - для каждого потенциального целевого формата проверяется:
    - наличие требуемых зависимостей по `requires*` полям,
    - текущее состояние `systemCapabilities` (см. 6.5).
- Форматы делятся на группы:
  - **Доступные**:
    - все зависимости присутствуют,
    - отображаются в основном списке, доступны для выбора.
  - **Условно недоступные (поиск совпал, но зависимость отсутствует)**:
    - показываются серым внизу списка с пометкой:
      - «Требуется LibreOffice»,
      - «Требуется FFmpeg»,
      - «Требуется Ghostscript»,
      - «Требуется pandoc/calibre»,
      - «Требуется libvips‑plugins (HEIC/RAW)»,
      - «Требуется Tesseract (OCR)»,
      - «Не поддерживается».
    - клик по такой строке не выбирает формат, а открывает модальное окно с инструкцией (если зависимость известна).
- При поиске по строке:
  - учитываются `name`, `extension`, `category`;
  - при совпадении, но невозможности конвертации (нет правила `compatibleWith`) формат отправляется в «неподходящие» в конце списка с пометкой «Не поддерживается для этого типа файла».

  - обработка EPS/AI;
  - flatten PDF.
- pandoc / calibre:
  - e‑book конвертация (EPUB, MOBI, AZW3, FB2);
  - Markdown → PDF/HTML/DOCX (pandoc).
- libvips‑plugins:
  - поддержка HEIC/HEIF;
  - поддержка RAW (CR2, NEF, ARW, DNG).
- Tesseract:
  - OCR по изображениям и PDF.
- Blender (headless, `blender --background --python`):
  - конвертация 3D‑моделей (STL/OBJ/GLTF/GLB/BLEND/FBX/3MF/PLY);
  - оптимизация сетки, рендер превью.
- CAD‑конвертер (ODA File Converter / LibreCAD / FreeCAD + Inkscape):
  - DWG/DXF/STEP/IGES → DXF/PDF/PNG/SVG;
  - проверяется через обнаружение бинаря и версию.
- 7-Zip / bsdtar (libarchive):
  - распаковка образов ISO/IMG/DMG;
  - распаковка и просмотр пакетов APK/DEB/RPM/MSI;
  - перепаковка RAR/7z при отсутствии собственных инструментов ОС.

### 6.4 История операций (history.json)

- Файл: `backend/history.json`.
- Структура:

```json
{
  "files": [
    {
      "id": "uuid",
      "originalName": "report.docx",
      "operation": "convert",
      "targetFormat": "pdf",
      "sizeBytes": 123456,
      "convertedAt": "2026-06-08T12:00:00Z",
      "downloadPath": "converted/report.pdf",
      "operationParams": {
        "quality": null,
        "crop": null,
        "filter": null,
        "password": null
      }
    },
    {
      "id": "uuid2",
      "originalName": "archive.zip",
      "operation": "extract_archive",
      "targetFormat": "folder",
      "sizeBytes": 987654,
      "convertedAt": "2026-06-09T10:30:00Z",
      "downloadPath": "converted/extracted_archive.zip",
      "operationParams": {
        "archiveFormat": "zip"
      }
    }
  ]
}
```

- Поле `operation` принимает значения:
  - `convert`,
  - `compress`,
  - `edit_image`,
  - `edit_video`,
  - `pdf_merge`, `pdf_split`, `pdf_protect`, `pdf_unlock`, `pdf_rotate`, `pdf_extract_images`, `pdf_flatten`, `pdf_crop`,
  - `extract_archive`, `create_archive`,
  - `ebook_convert`,
  - `vector_convert`,
  - `convert_3d`, `render_3d`,
  - `cad_convert`,
  - `format_code`, `minify_code`, `code_to_pdf`,
  - `convert_database`, `export_table`,
  - `extract_disk_image`,
  - `extract_app_package`,
  - `ocr_image`, `ocr_pdf`,
  - `video_to_gif`,
  - `batch`.
- `downloadPath`:
  - относительный путь к файлу/архиву в `./converted`;
  - для batch‑операций может ссылаться на ZIP‑архив с группой файлов.
- При запуске сервера:
  - `history.json` читается;
  - для каждой записи проверяется наличие файла по `downloadPath`;
  - записи без файлов удаляются и файл истории перезаписывается.

### 6.5 Системные зависимости (systemCapabilities)

- API: `GET /api/system/capabilities`.
- Пример ответа:

```json
{
  "ffmpeg": { "found": true, "version": "6.0" },
  "libreoffice": { "found": false, "version": null },
  "ghostscript": { "found": true, "version": "10.0" },
  "pandoc": { "found": false, "version": null },
  "calibre": { "found": false, "version": null },
  "libvipsPlugins": { "found": true, "hasHeic": true, "hasRaw": true },
  "tesseract": { "found": true, "version": "5.3" },
  "blender": { "found": false, "version": null },
  "cadConverter": { "found": false, "kind": null, "version": null },
  "sevenZip": { "found": true, "version": "23.01" },
  "bsdtar": { "found": true, "version": "3.5" }
}
```

- Логика проверки при старте сервера:
  - FFmpeg:
    - выполняется команда `ffmpeg -version` (или OS‑специфичный аналог);
    - в случае успеха `found = true`, `version` — первая строка версии.
  - LibreOffice:
    - выполняется `soffice --version` (или `libreoffice --version`);
    - `found = true` при успешном выполнении.
  - Ghostscript:
    - команда `gs --version`.
  - pandoc:
    - `pandoc --version`.
  - calibre:
    - `ebook-convert --version` или аналог.
  - libvipsPlugins:
    - проверяется, что `sharp.format.heif` и/или `sharp.format.raw` имеют флаг `input/output = true`;
    - при необходимости может выполняться тестовое чтение файла с диска (однократно при старте).
  - tesseract:
    - `tesseract --version`, если используется системный бинарь;
    - для Tesseract.js дополнительно может храниться состояние загрузки моделей (в отдельной структуре).
  - Blender:
    - `blender --version` (или поиск по типичным путям установки);
    - `found = true`, `version` — первая строка версии;
    - используется для конвертации 3D‑моделей и рендера превью в headless‑режиме.
  - CAD‑конвертер:
    - последовательно проверяются ODA File Converter (`ODAFileConverter --version`/наличие бинаря), LibreCAD, FreeCAD, Inkscape;
    - в ответе фиксируется `kind` (`oda`/`librecad`/`freecad`/`inkscape`/`null`) и `version`;
    - используется для DWG/DXF/STEP/IGES → DXF/PDF/PNG/SVG.
  - 7-Zip:
    - `7z` (Windows/Linux) или `7zz` (macOS);
    - `found = true`, `version` парсится из первой строки;
    - используется для распаковки RAR/7z, образов ISO/IMG/DMG и пакетов APK/DEB/RPM/MSI.
  - bsdtar (libarchive):
    - `bsdtar --version`;
    - резервный механизм для образов дисков и пакетов, когда 7-Zip недоступен.

- Кэширование:
  - результат проверки сохраняется в памяти;
  - повторная проверка выполняется по запросу `GET /api/system/capabilities?refresh=true`;
  - при refresh сервер повторно запускает все проверки.

- Использование на фронтенде:
  - для каждой записи `formats.json` сравниваются `requires*` с `systemCapabilities`;
  - недоступные форматы:
    - помечаются как заблокированные (серый цвет, текст «Требуется …»),
    - при клике открывают модальное окно с инструкцией;
  - при попытке использовать функциональность, которую нельзя определить только по формату (например, OCR):
    - фронт запрашивает `/api/system/capabilities`,
    - при отсутствии Tesseract — показывает модальное окно и не даёт запустить задачу.

## 7. АДАПТИВНОСТЬ И ПРОИЗВОДИТЕЛЬНОСТЬ

### 7.1 Мобильная версия (< 640 px)

- Общая компоновка:
  - все основные блоки располагаются вертикально друг под другом;
  - отступы по бокам — не менее 16 px.
- Зона загрузки файла:
  - ширина 100 % (минус боковые отступы);
  - высота адаптивная (подстраивается под содержимое);
  - кнопка «Выбрать файл» — полноширинная.
- Поле поиска целевого формата:
  - на всю ширину;
  - выпадающий список форматов занимает ширину контейнера, открывается вверх или вниз в зависимости от высоты экрана.
- Быстрые кнопки:
  - 2–3 кнопки в строку (flex‑сетка);
  - при недостатке места переносятся на следующую строку.
- Панель конвертации (ProgressPanel):
  - иконки исходного и целевого формата — вертикально (одна над другой);
  - стрелка направления может быть упрощена (вниз или↕);
  - прогресс‑бар — на всю ширину;
  - кнопки «Отмена» и «Скачать» — полноширинные, одна под другой (или рядом, если позволяет ширина).
- Страница истории:
  - каждая запись — вертикальный блок:
    - иконка исходного формата сверху или слева;
    - имя файла, тип операции, целевой формат, размер, дата — под иконкой;
    - кнопки «Скачать» и «Удалить» — под названием файла, на всю ширину.
- Страница форматов:
  - карточки форматов — по 2 в ряд;
  - поиск по форматам — на всю ширину, список/сетка форматов перестраивается под одну колонку при очень узких экранах.
- Вкладка Tools:
  - левая боковая панель инструментов (изображения, видео, PDF, архивы, e‑books, вектор, OCR, batch) заменяется:
    - либо на выпадающий список (`<select>`),
    - либо на горизонтальные табы с прокруткой (scrollable tabs) над рабочей областью;
  - сам контент выбранного под‑инструмента отображается под табами вертикально.

### 7.2 Отключение анимаций на слабых устройствах (алгоритм FPS‑теста)

#### 7.2.1 Механизм обнаружения

1. При загрузке страницы создаётся скрытый `<div>` с CSS‑анимацией изменения `opacity` (0 → 1 → 0 за 1 секунду, бесконечный цикл).
2. Запускается цикл `requestAnimationFrame`, счётчик кадров обнуляется, фиксируется стартовое время (`performance.now()`).
3. Через 2 секунды (по `setTimeout`) вычисляется средний FPS:
   - `avgFPS = frames / elapsedSeconds`.
4. Если вкладка неактивна (`document.visibilityState === 'hidden'`), тест откладывается:
   - подписка на `visibilitychange`,
   - запуск теста только когда вкладка становится `visible`.
5. Критерии включения режима низкой производительности:
   - `avgFPS < 30`,
   - или медиа‑запрос `prefers-reduced-motion: reduce`,
   - или `navigator.hardwareConcurrency < 4`.
6. При выполнении хотя бы одного условия:
   - устанавливается флаг `lowPerformance = true` в глобальном состоянии (например, в Zustand);
   - результат кэшируется (чтобы не гонять тест постоянно).
7. Повторный тест:
   - автоматически не запускается;
   - может быть запущен пользователем вручную из настроек (см. ниже);
   - не должен запускаться чаще 1 раза в 10 секунд.

#### 7.2.2 Изменения при `lowPerformance = true`

- Анимации Framer Motion:
  - отключаются (используются только статические состояния, без плавных переходов);
  - компоненты получают `reducedMotion`/`animate={false}`.
- Эффект Liquid Glass:
  - `backdrop-blur` заменяется на обычный полупрозрачный фон:
    - светлая тема: `background: rgba(255,255,255,0.8)`;
    - тёмная тема: `background: rgba(31,41,55,0.8)`;
  - тени могут быть упрощены (меньше размытия).
- Боковые градиенты:
  - отключается анимация движения/изменения интенсивности;
  - остаются статичные радиальные градиенты по краям.
- Анимация движущейся полосы завершения:
  - полностью отключается;
  - вместо этого используется мгновенный переход к финальному статическому градиенту.
- Hover‑эффекты:
  - остаются, но без плавных переходов (`transition: none`);
  - изменение цвета/тени происходит мгновенно.
- Ошибки:
  - тряска экрана (`.shake`) не применяется;
  - остаётся только лёгкий glitch‑эффект текста ошибки (может быть упрощён).
- Прогресс‑бар:
  - частота визуального обновления ограничивается 1 разом в секунду;
  - если WebSocket шлёт события чаще, фронтенд дросселирует обновления состояния (throttle).

#### 7.2.3 Ручное управление производительностью

- В правой части App Bar (рядом с переключателем языка) находится иконка «Настройки» (шестерёнка).
- При клике открывается выпадающее меню (или модальное окно) с опциями:
  - «Автоопределение производительности» — переключатель (по умолчанию включён).
  - «Отключить все анимации» — принудительно устанавливает `lowPerformance = true` и блокирует авто‑FPS‑тест.
  - «Включить все анимации» — принудительно устанавливает `lowPerformance = false`, игнорируя результаты теста.
- Выбранный режим сохраняется в `localStorage`:
  - при старте приложения сначала читается сохранённое значение;
  - если пользователь ничего не задавал явно, используется авто‑режим.

### 7.3 Ленивая загрузка (Lazy Loading)

- Иконки форматов:
  - загружаются по требованию (например, через динамический импорт компонент/иконок);
  - иконки, не попадающие в видимую область, могут подгружаться при появлении в viewport (через Intersection Observer).
- Компоненты редакторов (Tools):
  - `ImageEditor`, `VideoEditor`, `PdfTools`, компоненты для архивов, e‑books, OCR — грузятся только при выборе соответствующего под‑инструмента;
  - Next.js dynamic import (`next/dynamic`) без SSR для тяжёлых компонентов.
- Тяжёлые библиотеки:
  - `puppeteer`, `ffmpeg`, `sharp` используются только на бэкенде;
  - на фронтенде отсутствуют WebAssembly‑версии тяжёлых конвертеров (чтобы не увеличивать бандл).
- Превью изображений:
  - для отображения предпросмотра используется уменьшенная копия, полученная на клиенте через Canvas API;
  - оригинал файла не рендерится в `<img>` в полном разрешении без необходимости.
- Асинхронная загрузка функционала:
  - код для работы с WebSocket, менеджер задач, дополнительная логика истории могут быть вынесены в отдельные чанки;
  - подключаются при первом запуске операции (например, при клике «Конвертировать» или переходе на `/history`).
- Общие принципы:
  - критический интерфейс (зона загрузки, базовая навигация) загружается первым;
  - второстепенный функционал (How it works, About, часть Tools) загружается лениво;
  - для изображений и иллюстраций используется `loading="lazy"` и responsive‑подход (разные размеры под разные экраны).

## 8. ОБРАБОТКА ОШИБОК

### 8.1 Общие принципы

- Все ошибки, возвращаемые сервером, имеют структурированный JSON‑ответ:
  - `{ error: true, code: string, message: string, details?: any }`.
- На клиенте ошибки обрабатываются централизованно:
  - через axios‑interceptors для HTTP‑запросов;
  - через единый обработчик WebSocket‑сообщений со `status: 'failed'`.
- При критических ошибках (кроме явной отмены пользователем) запускается визуальный эффект **glitch + тряска экрана** (если не установлен `lowPerformance = true`).

### 8.3 Логирование ошибок на сервере

- Все ошибки уровня `error` логируются в файл `backend/error.log`.
- Формат записи:
  - `[ISO дата время] [HTTP метод] [URL] [код ошибки] [message] [stack trace (если есть)]`.
- В локальной версии:
  - ротация лог‑файла автоматически не выполняется;
  - пользователь может удалить/очистить файл вручную при необходимости.
- Для особо тяжёлых ошибок (падение внешнего процесса FFmpeg, LibreOffice, Ghostscript, pandoc, calibre, Blender, CAD‑конвертера, 7-Zip/bsdtar):
  - логируются команды запуска и коды возврата процессов.

---

### 8.4 Поведение при ошибках в пакетном режиме (Batch)

- При batch‑операции (до 50 файлов):
  - если один из файлов не удалось обработать (любая ошибка из таблицы выше), вся пакетная задача помечается как `failed`;
  - пользователю отображается сообщение вида:
    - «Ошибка при обработке файла „имя_файла“: <текст ошибки>. Пакетная обработка остановлена.»
- Архив с результатами:
  - при ошибке хотя бы по одному файлу итоговый ZIP‑архив не создаётся;
  - уже обработанные файлы в `./tmp` и `./converted/batch_<batchJobId>/` удаляются.
- Пользовательский интерфейс:
  - панель прогресса для batch показывает статус `failed`;
  - кнопка «Попробовать снова» сбрасывает состояние и позволяет выбрать файлы заново или убрать проблемный файл и запустить пакет снова.
- В будущем (опциональный режим):
  - возможно добавление настройки «Пропускать проблемные файлы и продолжать обработку остальных»,
  - в текущей версии MVP поведение строгое: ошибка одного файла = ошибка всего пакета.

## 10. ЛОКАЛИЗАЦИЯ (i18n)

### 10.1 Общие принципы

- Библиотека: `next-intl` (поддержка App Router, динамическая загрузка сообщений).
- Локали: минимум `ru` и `en`.
- Язык по умолчанию:
  - определяется по заголовку `Accept-Language` браузера;
  - если первый подходящий язык не `ru` и не `en`, используется `en`.
- Локализованная маршрутизация:
  - все страницы доступны с префиксом `/[locale]`, например:
    - `/ru/convert`, `/ru/compress`, `/ru/tools`, `/ru/history`, `/ru/formats`, `/ru/how-it-works`, `/ru/about`;
    - `/en/convert`, `/en/compress`, `/en/tools`, `/en/history`, `/en/formats`, `/en/how-it-works`, `/en/about`.
  - в папке `app/[locale]/` находятся все layout'ы и страницы (см. структуру проекта).

### 10.2 Переключение языка

- Кнопка «RU»/«EN» в App Bar:
  - реализована через `next-intl` навигацию (`useRouter`, `usePathname` или `createSharedPathnamesNavigation`);
  - при клике выполняется редирект на ту же страницу с новым префиксом локали.
- Поведение при переключении:
  - состояние текущей страницы (выбранный файл, прогресс, целевой формат) в MVP не сохраняется и сбрасывается при перезагрузке;
  - опционально, в будущем можно сохранять выбранный файл и формат в `localStorage` на вкладке Convert.
- Текущая локаль:
  - хранится в сегменте URL (`[locale]`);
  - дополнительно может записываться в `localStorage` для выбора по умолчанию при следующем визите.

  - сервер отдаёт код ошибки (`code`), фронтенд выбирает нужную строку по ключу.
- Умные подсказки по форматам:
  - сценарии «PNG → WebP (compress for web)» и т. д. локализуются через словарь;
  - текст подсказки формируется из шаблонов (форматы и стрелка можно не переводить).
- Названия форматов и категорий:
  - отображаемое имя (например, «EPUB E-book», «ZIP Archive») берётся из i18n;
  - внутренние поля `extension`/`category` в `formats.json` не локализуются.
- Статусы операций:
  - «Конвертация…», «Сжатие…», «Распаковка архива…», «Распознавание текста…» выводятся через `status.*`.
- Инструкции в модальных окнах:
  - тексты по установке FFmpeg, LibreOffice, Ghostscript, pandoc/calibre, libvips‑plugins, Tesseract.js.

### 10.5 Форматирование дат, чисел и множественного числа

- Даты:
  - форматирование через `Intl.DateTimeFormat` с учётом текущей локали;
  - пример (псевдо): `new Intl.DateTimeFormat(locale, { dateStyle: 'medium', timeStyle: 'short' })`.
- Размеры файлов:
  - форматирование чисел через `Intl.NumberFormat` (например, `1,23 MB` или `1,23 МБ` в зависимости от локали);
  - единицы (KB/MB/GB) задаются строками в словаре.
- Плюрализация:
  - для сообщений наподобие «1 файл обработан», «2 файла обработано» используются plural‑формы через `next-intl` (`t('key', {count})`);
  - в JSON можно использовать ключи вида `processedFiles_one`, `processedFiles_few`, `processedFiles_many`, `processedFiles_other` для русского и `processedFiles_one`, `processedFiles_other` для английского.

### 10.6 Fallback и поведения при отсутствии перевода

- Базовый язык: `en`.
- Если ключ отсутствует в текущей локали:
  - используется значение из `en.json` (fallback);
  - в dev‑режиме возможно логирование отсутствующих ключей в консоль для упрощения отладки.
- Если локаль не указана в URL:
  - middleware `next-intl` перенаправляет на `/[defaultLocale]` (обычно `/en`), либо на результат автоопределения по `Accept-Language`.

**Итого ориентировочно:** ~20 недель чистой разработки (без учёта возможных параллельных работ и буфера на риски). Этапы 1–9 формируют MVP; этапы 10–13 — расширение базового функционала; этапы 14–17 — профессиональные форматы (3D, CAD, code, DB, disk, apps).  
Некоторые этапы (например, 10, 11 и 14) могут выполняться параллельно отдельными разработчиками при наличии ресурсов. Этап 15 может быть разбит на два параллельных потока: 3D и CAD.

## ПРИЛОЖЕНИЕ: ИЗВЕСТНЫЕ РИСКИ И МЕРЫ СМЯГЧЕНИЯ

| Риск | Вероятность | Смягчение |
|------|-------------|-----------|
| React Compiler нестабилен в production | Средняя | Отключён в production-сборке, используется только в dev-окружении для экспериментов. При проблемах легко отключается флагом в конфигурации сборки. |
| Turbopack не готов для production | Низкая | Не используется в production; для прод-сборок применяется стабильный `next build` на основе Webpack. Turbopack допускается только для `next dev`. |
| Один экземпляр Puppeteer может привести к утечке памяти | Низкая | Явное закрытие страниц после каждой задачи; перезапуск браузера раз в N задач (например, 100 конвертаций) или после интервала простаивания (10 минут). Мониторинг памяти и логирование аномалий. |
| Системный FFmpeg отсутствует | Средняя | При старте проверяется наличие FFmpeg; при отсутствии операции, требующие FFmpeg, помечаются недоступными. Пользователю показывается модальное окно с понятной инструкцией по установке FFmpeg под Windows/macOS/Linux и ссылкой на официальный сайт. |
| FPS-тест в неактивной вкладке даёт ложный низкий FPS | Низкая | Перед запуском теста проверяется `document.visibilityState`; если вкладка `hidden`, тест откладывается до события `visibilitychange` и запускается только в активной вкладке. |
| Большие таблицы (>100 МБ) зависают в `worker_thread` | Низкая | Для таблиц добавляется прогресс-бар (эмуляция/чекап шагов), вводится верхний лимит размера (например, 500 МБ); при превышении — раннее предупреждение и отказ от обработки, без зависания процесса. |
| LibreOffice не найден на нестандартных путях Windows | Средняя | Проверяются типовые пути установки LibreOffice. При отсутствии в PATH и стандартных каталогах показывается инструкция по добавлению LibreOffice в PATH и указанию пользовательского пути в конфиге (опционально). |
| Blender не установлен или не доступен в PATH | Средняя | При старте проверяется `blender --version`; при отсутствии 3D‑операции через Blender помечаются недоступными. Лёгкие STL/OBJ/GLTF конвертации (three.js) работают без Blender. Модальное окно с инструкцией по установке. |
| CAD‑конвертер (ODA/LibreCAD/FreeCAD/Inkscape) отсутствует или не поддерживает формат | Средняя | Последовательно проверяются доступные CAD‑инструменты; при отсутствии всех — CAD‑операции помечаются недоступными. Модальное окно с инструкцией и ссылками. Для保护的 файлов DWG показывается предупреждение. |
| 7-Zip/bsdtar отсутствуют на целевой ОС | Низкая | На Windows 7-Zip обычно установлен; на macOS/Linux проверяется наличие `7zz`/`7z` и `bsdtar`. При отсутствии обоих — образы дисков и пакеты приложений помечаются недоступными. Модальное окно с инструкцией. |
| Blender headless занимает много ОЗУ при конвертации тяжёлых 3D‑сцен (>100 МБ) | Средняя | Мониторинг потребления памяти; при превышении порога (например, 80 % доступной ОЗУ) — отказ с `INSUFFICIENT_RESOURCES`; прогресс через heartbeats; кэширование инстанса Blender для серии задач. |
| JSON/YAML/XML файл свыше 500 МБ вызывает OOM в worker_thread | Низкая | Для code/data операций вводится лимит 500 МБ; большие файлы обрабатываются потоково (streaming parse для JSON/YAML) или отказ с `FILE_TOO_LARGE`. |
